
# Документация API: WP Admin Assistant (Бэкенд)

Этот проект представляет собой модульный монолит на базе Flask, предоставляющий REST API для инструментов автоматизации (на данный момент реализован модуль обработки `.docx` файлов).

## 1. Запуск тестового окружения

Для запуска проекта на локальной машине необходимо настроить виртуальное окружение и установить зависимости.

### Шаг 1: Создание виртуального окружения

Откройте терминал в корневой папке проекта и выполните команду:

**Для Windows:**

```bash
python -m venv venv
venv\Scripts\activate

```



### Шаг 2: Установка зависимостей

Убедитесь, что виртуальное окружение активировано (в терминале должно появиться `(venv)`), и установите библиотеки из файла `requirements.txt`:

```bash
pip install -r requirements.txt

```

### Шаг 3: Запуск сервера

Запустите главный файл приложения:

```bash
python app.py

```

Если всё прошло успешно, сервер запустится на порту `5001`. Доступные локальные адреса:

* API: `http://127.0.0.1:5001`
* Документация Swagger: `http://127.0.0.1:5001/apidocs/`

---

## 2. Использование интерактивной документации (Swagger UI)

Swagger UI — это инструмент для фронтенд-разработчиков и тестировщиков, позволяющий отправлять реальные запросы к API прямо из браузера без использования сторонних программ вроде Postman.

### Как протестировать загрузку документа:

1. Перейдите по адресу: [http://127.0.0.1:5001/apidocs/](http://127.0.0.1:5001/apidocs/).
2. Найдите секцию **Doc to Blocks Tool** и кликните на эндпоинт `POST /api/tools/doc-to-blocks/`.
3. В правом верхнем углу открывшегося блока нажмите кнопку **Try it out** (Опробовать).
4. В появившемся поле `file` нажмите **Выберите файл** и загрузите любой тестовый документ формата `.docx` с вашего компьютера.
5. Нажмите большую синюю кнопку **Execute** (Выполнить).
6. Прокрутите чуть ниже к блоку **Responses** -> **Server response**. Там вы увидите реальный JSON-ответ сервера и HTTP статус-код (например, `200`).

## 3. Внутренняя логика: функция `convert_doc_to_wp_tags`

Данная функция располагается в модуле сервиса (`service.py`) и инкапсулирует бизнес-логику обработки Word-документов.

> **Примечание:** На текущем этапе разработки (несмотря на название функции) она работает как **счетчик символов** документа, а не как генератор HTML-тегов.

### Описание работы

Функция принимает бинарный объект файла, парсит его с помощью библиотеки `python-docx`, извлекает весь текст по абзацам и подсчитывает количество символов.

### Сигнатура

```python
def convert_doc_to_wp_tags(file) -> str

```

* **Аргументы:**
* `file`: Файловый объект `.docx` (полученный из `request.files['file']`).


* **Возвращаемое значение:**
* Успех: Форматированная строка с результатами подсчета. Пример: `количество символов с пробелами - 150, Количество символов без пробелов - 130`
* Ошибка: В случае поврежденного файла или неверного формата возвращает строку с описанием исключения: `Ошибка при чтении документа: <текст_ошибки>`.

## 4. Справочник API Эндпоинтов

### 4.1. Health Check

* **Метод:** `GET`
* **URL:** `/health`
* **Описание:** Проверка работоспособности сервера.
* **Ответ (200 OK):**
```json
{
  "message": "WP Admin Assistant API is running",
  "status": "ok"
}

```



### 4.2. Конвертер Word-документов

* **Метод:** `POST`
* **URL:** `/api/tools/doc-to-blocks/`
* **Content-Type:** `multipart/form-data`
* **Параметры тела запроса:**
* `file` (обязательный): Файл формата `.docx`.



**Пример успешного ответа (200 OK):**

```json
{
  "html": " количество символов с пробелами - 1245, Количество символов без пробелов - 1020"
}

```

**Пример ответа с ошибкой валидации (400 Bad Request):**

```json
{
  "error": "No file provided"
}

```
